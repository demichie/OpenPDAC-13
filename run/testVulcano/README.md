# testVulcano

`testVulcano` is an OpenPDAC demonstration case for simulating a phreatic explosion over real topography. The case follows an automated four-stage workflow:

1. mesh generation and deformation onto the topography;
2. field initialization;
3. transient multiphase simulation with OpenPDAC;
4. post-processing and production of diagnostic outputs, maps, and visualizations.

The workflow is designed to be reproducible and modular. Each stage can be run independently with the `01_*`--`04_*` scripts, or the full sequence can be executed with `Allrun`.

## Requirements

Before running the case, make sure that the following are available:

- an OpenFOAM version compatible with OpenPDAC and with the utilities used by this case (`blockMesh`, `decomposePar`, `checkMesh`, `topoSet`/`createZones`, `topoGrid`, `setFields`, `foamVTKSeries`);
- the OpenPDAC solver specified in the `system/controlDict.*` files;
- a Conda environment named `OpenPDACconda`, used by the Python preprocessing and post-processing scripts;
- the Python packages required by the scripts, in particular `numpy`, `pandas`, `matplotlib`, `scipy`, `pyvista`, `rasterio`, and the other workflow-specific dependencies;
- `ffmpeg`, if MP4 videos are to be generated by `plotIso.py`;
- a DEM in ESRI ASCII format (`.asc`) inside `constant/DEM/`, or a valid DEM path passed to `01_run_meshing.sh`.

## Quick start

To run the full case with the default DEM:

```bash
./Allrun
```

To run the workflow stages manually:

```bash
./01_run_meshing.sh [DEM_ASC_FILE]
./02_run_fieldInitialization.sh
./03_run_simulation.sh
./04_run_postprocessing.sh
```

If `DEM_ASC_FILE` is not provided, the meshing script uses `dsm_vulc_5m.asc` as the default input file.

## Workflow structure

### `01_run_meshing.sh` — mesh generation

This script prepares the geometry and builds the computational mesh. In particular, it:

1. moves to the case directory, so that the script can be launched from any location;
2. loads the standard OpenFOAM `RunFunctions`;
3. activates the `OpenPDACconda` Conda environment;
4. cleans the case by calling `./Allclean`;
5. runs `smoothCraterArea.py` on the selected DEM, producing a modified DEM in the crater area;
6. copies `system/controlDict.init` to `system/controlDict`;
7. generates the precursor mesh with `blockMesh`;
8. recreates the `0` directory from `org.0`;
9. decomposes the domain with `decomposePar`;
10. checks the initial mesh in parallel with `checkMesh` and stores the first log as `log.checkMesh0`;
11. applies topological changes with `createZones`;
12. deforms the mesh onto the topography with `topoGrid`;
13. runs a second `checkMesh` on the deformed mesh.

At the end of this stage, the mesh is available in the `processor*` directories. The decomposed mesh can be inspected in ParaView by opening the case `.foam` file.

### `02_run_fieldInitialization.sh` — field initialization

This script prepares the initial fields before the main simulation. It requires the mesh to have already been generated. The main operations are:

1. copy the initialization dictionaries:
   - `system/controlDict.init` -> `system/controlDict`;
   - `system/fvSolution.init` -> `system/fvSolution`;
   - `constant/cloudProperties.init` -> `constant/cloudProperties`;
2. run the solver indicated by `getApplication` in parallel, in order to initialize the atmospheric profile and the base fields;
3. store the initialization log as `log.foamRun0`;
4. run `setFields` in parallel to impose the problem-specific initial regions, such as the explosive source inside the crater.

After this stage, the initialized fields are stored in the `processor*` directories and can be inspected in ParaView.

### `03_run_simulation.sh` — main simulation

This script runs the transient simulation. In particular, it:

1. copies the run-stage dictionaries:
   - `system/controlDict.run` -> `system/controlDict`;
   - `system/fvSolution.run` -> `system/fvSolution`;
   - `constant/cloudProperties.run` -> `constant/cloudProperties`;
2. launches the OpenPDAC solver indicated by `getApplication` in parallel;
3. after the run is complete, analyzes `log.foamRun` with `analyzeLog.py`.

The `analyzeLog.py` script extracts numerical diagnostics from the solver log and produces a CSV table and PNG/PDF figures in the `log_analysis/` directory. The diagnostics include execution time, `deltaT`, Courant numbers, pressure residuals, pressure and temperature extrema, volume fractions, granular temperature, and PIMPLE-iteration information.

### `04_run_postprocessing.sh` — post-processing

This script activates the `OpenPDACconda` Conda environment and generates post-processing outputs from the data written during the simulation.

In the current version of the script:

1. previous ballistic raster maps in `postProcessing/raster_maps/ballistics/` are removed;
2. the call to `plotBallistics.py` is prepared but commented out;
3. previous flow raster maps in `postProcessing/raster_maps/flow/` are removed;
4. the call to `createMaps.py` is prepared but commented out;
5. previous PNG frames in `postProcessing/frames_png/` are removed;
6. `plotIso.py --np 10 --fr 10 --ps 50` is executed, with output redirected to `log.plotiso`;
7. `foamVTKSeries` is executed;
8. `VTKballistics_series.py` is executed to build a VTK series for ballistic particles;
9. the Conda environment is deactivated.

To also generate ballistic maps and flow raster maps, uncomment the following lines:

```bash
python3 plotBallistics.py > log.plotBallistics
python3 createMaps.py -np 10 > log.createMaps
```

## Utility scripts

### `Allrun`

`Allrun` automatically executes the full workflow and stops at the first error through `set -e`:

1. `01_run_meshing.sh`;
2. `02_run_fieldInitialization.sh`;
3. `03_run_simulation.sh`;
4. `04_run_postprocessing.sh`.

This is the recommended command for reproducing the complete case with the default settings.

### `Allclean`

`Allclean` cleans the case and restores it to an initial state. The script:

1. moves to the case directory;
2. loads the standard OpenFOAM `CleanFunctions`;
3. runs `cleanCase`, removing standard logs, time directories, and temporary files;
4. removes the `0` directory;
5. removes `log_analysis/`;
6. removes files produced by DEM preprocessing:
   - `constant/DEM/output_file.asc`;
   - `constant/DEM/modified_points.png`;
7. removes dictionaries copied by the workflow scripts:
   - `system/controlDict`;
   - `system/fvSolution`;
   - `constant/cloudProperties`.

Template files such as `controlDict.init`, `controlDict.run`, `fvSolution.init`, `fvSolution.run`, `cloudProperties.init`, and `cloudProperties.run` are not removed.

## Main outputs

### Mesh and fields

After `01_run_meshing.sh` and `02_run_fieldInitialization.sh`, the `processor*` directories contain the decomposed mesh and initialized fields. The main logs are:

- `log.blockMesh`;
- `log.decomposePar`;
- `log.checkMesh0`, corresponding to the mesh check before deformation;
- `log.checkMesh`, corresponding to the final mesh check;
- `log.createZones`;
- `log.topoGrid`;
- `log.foamRun0`, corresponding to the initialization stage;
- `log.setFields`.

### Simulation

The main simulation produces the `log.foamRun` log and OpenFOAM data in the `processor*` directories, together with function-object outputs in `postProcessing/`.

The automatic log diagnostics produce:

- `log_analysis/log_diagnostics.csv`;
- PNG/PDF figures covering performance, time-step/Courant diagnostics, residuals, pressure, temperature, volume fractions, granular temperature, and PIMPLE iterations.

### Post-processing

The post-processing stage can produce several types of output:

- MP4 videos in `postProcessing/videos_mp4/`, generated by `plotIso.py`, with top-down, isometric, and orbital views of isosurfaces and Lagrangian particles;
- temporary frames in `postProcessing/frames_png/`, retained only if video creation fails;
- VTK/ParaView series generated by `foamVTKSeries`;
- `particles_*.vtk` files and `particles.vtk.series` in `postProcessing/cloudInfo1/`, generated by `VTKballistics_series.py` to visualize the evolution of ballistic particles in ParaView;
- if `createMaps.py` is enabled: flow raster maps in `postProcessing/raster_maps/flow/`, in PNG and ESRI ASCII (`.asc`) format, obtained by interpolating `alpha*` quantities onto a regular grid;
- if `plotBallistics.py` is enabled: ballistic impact density/probability maps in `postProcessing/raster_maps/ballistics/`, in PNG and ESRI ASCII (`.asc`) format.

The `.asc` raster products can be imported into GIS software, while `.vtk.series` files can be opened directly in ParaView.

## Operational notes

- The number of processors used by OpenFOAM utilities is controlled by `system/decomposeParDict`.
- In `04_run_postprocessing.sh`, the `--np 10` option of `plotIso.py` and `createMaps.py` controls Python-side post-processing parallelism, not necessarily the number of OpenFOAM subdomains.
- Some post-processing products depend on the function objects enabled during the simulation. If expected folders are missing in `postProcessing/`, check the dictionaries in `system/` and the `log.foamRun` log.
- To visualize particles in ParaView, open `postProcessing/cloudInfo1/particles.vtk.series` after running `VTKballistics_series.py`.
